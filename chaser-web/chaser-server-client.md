# CHaser サーバ／クライアント実装仕様書（Ver.2.0 準拠）

本書は、全国情報技術教育研究会が公開している「CHaser ルールブック Ver.2.0」および CHaser クライアントの公開ソース（Ruby / Python）をもとに、CHaser のサーバおよびクライアントを独自実装するために必要な技術情報をまとめたものである。  

---

## 1. 用語

- **サーバ**  
  競技サーバ。マップ・ターン数・アイテム取得数・勝敗などを管理し、クライアントからの行動命令を実行して結果（周囲情報）を返す。  

- **クライアント**  
  競技者が作成する自律行動プログラム。Cool（先攻）または Hot（後攻）としてサーバに接続し、サーバから受け取る周囲情報にもとづき、次の行動を決定して送信する。  

- **Cool (C)** / **Hot (H)**  
  先に接続するクライアントを Cool（先攻）、後から接続するクライアントを Hot（後攻）と呼ぶ。  

- **ターン**  
  「C が 1 回行動し、その後 H が 1 回行動する」までを 1 ターンと呼ぶ。野球の「1 回表・裏」に相当する。  

- **ラウンド**  
  ターンを重ねて勝敗が決まるまでを 1 ラウンドと呼ぶ。  

- **フィールド（マップ）**  
  C/H やアイテム・ブロックが配置された 2 次元のマス目。構成は実行委員会が決定し、マップごとにターン制限が設定される。  

---

## 2. ゲーム仕様

### 2.1 マップの構成

マップは 2 次元の格子であり、各マスには以下のいずれかのパーツが存在する。  

| キャラクタ番号 | 意味                     |
|----------------|--------------------------|
| 0              | 床（何もない）           |
| 1              | キャラクタ（C または H） |
| 2              | ブロック                 |
| 3              | アイテム（ハート）       |

- マップサイズや外周（壁）の扱いはサーバのマップデータに依存し、ルールブックには最大サイズ等の規定はない。  

### 2.2 クライアントの行動

クライアントがサーバに指示できる行動は 4 系列・各 4 方向の計 16 種類である。  

- **walk 系**（自分を 1 マス移動）
  - `walkRight` / `walkLeft` / `walkUp` / `walkDown`
- **look 系**（指定方向の 3×3 周囲情報を取得）
  - `lookRight` / `lookLeft` / `lookUp` / `lookDown`
- **search 系**（指定方向の直線 9 マスの情報を取得）
  - `searchRight` / `searchLeft` / `searchUp` / `searchDown`
- **put 系**（指定方向の隣接マスにブロックを置く）
  - `putRight` / `putLeft` / `putUp` / `putDown`

これらは通信プロトコル上は 2 文字コマンドで表現される（4 章参照）。

### 2.3 勝敗条件

ルールブックで定義されている公式の勝敗条件は次の通り。  

#### 2.3.1 勝ち

以下の場合、そのプレイヤーの勝ちとなる。

1. **相手の上にブロックを置いたとき**  
   - `put` によって敵キャラクタのいるマスにブロックを設置できた場合。  
   - この場合、取得アイテム数は勝敗に影響しない。

2. **上記 1 が発生しないままターン制限に達した場合**  
   - 取得したアイテム数が多い方が勝ち。  
   - アイテム数が同数なら引き分け。

#### 2.3.2 負け（アイテム数にかかわらず即敗北）

次の状態になった側は、勝敗 1・2 にかかわらず負け。

3. 相手にブロックを置かれたことにより、自分の上下左右 4 方向がブロックに囲まれた。  
4. 自分でブロックを置いた結果、自分の上下左右 4 方向がブロックに囲まれた。  
5. 自分の行動（walk）で、自分がブロック上のマスに移動した。  

#### 2.3.3 アイテム取得時の自動ブロック

- アイテムを取得すると、**アイテムを取る直前にいたマスに自動的にブロックが置かれる**。  
- このブロック設置により上記 1〜5 の条件を満たした場合は、その条件に従って勝敗を決定する。

#### 2.3.4 引き分け

ルールブックで明記されている引き分け条件は次の通り。  

1. `put` により相手にブロックを置くことには成功したが、その結果、自分がブロックに囲まれた（2.3.2 の形）場合。  
2. `put` によるブロック設置で勝敗がつかず、ターン終了時点でアイテム数が同数だった場合。

## 3. 通信仕様（TCP）

### 3.1 通信方式とポート

- 通信方式: TCP
- サーバソケット番号（ポート番号）  
  - Cool: `40000`
  - Hot: `50000`

1 ラウンドごとにクライアントはサーバに接続する。接続制限時間や対戦運用ルールは大会運営側の規定による（実装上はサーバ側の接続タイムアウトとして実現可能）。  

### 3.2 接続から切断までのシーケンス概要

ルールブック「通信の仕様」に規定された手順は次の通り。  

1. **TCP 接続確立**  
   クライアントはサーバ（Cool/Hot 用ポート）へ TCP 接続する。

2. **チーム名送信**  
   - 接続直後にチーム名（文字列）を送信する。
   - 有効な文字数は以下の通り（追加の文字は無視される）:
     - 全角: 先頭 4 文字
     - 半角: 先頭 8 文字
   - 空文字列 `""` を送信した場合、チーム名は自動的に
     - 先攻: `"Cool"`
     - 後攻: `"Hot"`  
     となる。  

3. **サーバからの準備通知**  
   - サーバは `"@"` を送信し、クライアントはこれを受信する。  

4. **`getReady`**  
   - クライアントは文字列 `"gr"` に CRLF（`\r\n`）を付けて送信する。  
   - サーバは「制御情報＋周囲情報 10 文字」を送信する（例 `"1000000000"`）。  

5. **行動コマンド送信**  
   - クライアントは行動を表す 2 文字のメソッド名＋CRLF の 4 バイトを送信する（例 `"wr\r\n"`）。  
   - サーバはコマンドを実行し、再度「制御情報＋周囲情報 10 文字」を送信する。

6. **ターン終了通知**  
   - クライアントは `"#"` に CRLF を付けて送信する（例 `"#\r\n"`）。  

7. **以降のターン**  
   - 制御情報が `0` になるまで、サーバとクライアントは 3〜6 を繰り返す。  
   - C と H の行動は交互に行われ、サーバ側でターン数を管理する。  

### 3.3 コマンド一覧（クライアント → サーバ）

ルールブックに定義されているコマンドと方向の対応は以下の通り。  

| 系列       | 右 (`Right`) | 左 (`Left`) | 上 (`Up`) | 下 (`Down`) |
|------------|--------------|------------|----------|------------|
| **walk 系**   | `wr`         | `wl`       | `wu`     | `wd`       |
| **look 系**   | `lr`         | `ll`       | `lu`     | `ld`       |
| **search 系** | `sr`         | `sl`       | `su`     | `sd`       |
| **put 系**    | `pr`         | `pl`       | `pu`     | `pd`       |

クライアントは **必ず 2 文字のコマンド＋CRLF**（合計 4 バイト）を送信しなければならない。有効でない文字列を送信した場合、そのクライアントの負けとなる。

## 4. データフォーマット（制御情報＋周囲情報）

### 4.1 サーバからの 10 文字データ

サーバが `getReady` またはコマンド実行後に送信するデータは、**10 文字の数字文字列**である。  

- 例: `"1000000000"`

Python クライアント例では、このデータを受信後に次のように解釈している。  

```python
responce = self.client.recv(4096)[0:11].decode("utf-8")
if responce[0] == '1':
    return [int(x) for x in responce[1:10]]
elif responce[0] == '0':
    raise OSError("Game Set!")
````

この実装から、以下の点が読み取れる。

* 先頭 1 文字（`responce[0]`）が **制御情報**。
* 後続 9 文字（`responce[1:10]`）が **周囲情報**。
* 制御情報が `'1'` の場合にのみゲーム続行として解釈し、`1..9` を整数配列に変換する。
* 制御情報が `'0'` の場合はゲーム終了扱い。

**仕様としてまとめると:**

* **制御情報（index 0）**
  * `'1'`: ラウンド継続
  * `'0'`: ラウンド終了
  * それ以外の値は公式には定義されていない（サーバ／クライアント実装では `0/1` のみを使用する前提で設計すること）。
* **周囲情報（index 1〜9）**
  * 3×3 の周辺マスの状態を表す 9 文字。
  * 各文字は `0`〜`3` のいずれかで、マップ上のキャラクタ番号と同じ意味をもつ（2.1 参照）。

### 4.2 3×3 周囲情報のインデックス

Ruby マニュアルでは、サーバから取得した 9 文字の周囲情報を次のようにマッピングして利用している。

* クライアント側では通常、制御情報を除いた 9 要素を `info[0]`〜`info[8]` として扱う。

|  行\列  |     L     |     C     |     R     |
| :---: | :-------: | :-------: | :-------: |
| **U** | `info[0]` | `info[1]` | `info[2]` |
| **C** | `info[3]` | `info[4]` | `info[5]` |
| **D** | `info[6]` | `info[7]` | `info[8]` |

Ruby マニュアルでは、これに対応するシンボリック定数も定義されている。

| 位置 | 添字 | 定数名          |
| -- | -- | ------------ |
| 左上 | 0  | `UP_LEFT`    |
| 上  | 1  | `UP`         |
| 右上 | 2  | `UP_RIGHT`   |
| 左  | 3  | `LEFT`       |
| 中央 | 4  | `CENTER`     |
| 右  | 5  | `RIGHT`      |
| 左下 | 6  | `DOWN_LEFT`  |
| 下  | 7  | `DOWN`       |
| 右下 | 8  | `DOWN_RIGHT` |

### 4.3 周囲情報に含まれる値

周囲情報 `info[i]` に入る値と意味は、マップのキャラクタ番号に対応して次の 4 種類である。

| 値 | 定数名（Ruby 例）  | 意味              |
| - | ------------ | --------------- |
| 0 | `TYPE_FLOOR` | 床（何もなし）         |
| 1 | `TYPE_ENEMY` | キャラクタ（C または H）※ |
| 2 | `TYPE_BLOCK` | ブロック            |
| 3 | `TYPE_ITEM`  | アイテム（ハート）       |

※`TYPE_ENEMY` という名前だが、自分／敵の区別は付かない（中央 `info[4]` は常に自分の位置で 1 になる実装が多い）。Ruby マニュアルでも `info[1], info[3], info[5], info[7]` を「敵がいるかどうか」の判定に使用している。

## 5. クライアント実装仕様

### 5.1 接続〜終了までのアルゴリズム

1. **接続**
   * TCP ソケットを作成し、ユーザ入力等から得た IP アドレス・ポート（Cool:40000 / Hot:50000）へ接続する。
2. **チーム名送信**
   * チーム名文字列をサーバへ送信する（改行の有無は公式仕様に明記されていないが、公開クライアントでは CRLF を付与して送信している）。
3. **ゲームループ**
   * `while True`:
     1. `getReady()` を呼ぶ
        * サーバから `"@"` を受信する。
        * `"gr\r\n"` を送信する。
        * 制御＋周囲 10 文字を受信し、制御情報が `0` ならループを抜ける。
     2. 周囲情報にもとづき行動を決定する。
     3. 対応するコマンド（例: `"wr\r\n"`）を送信する。
     4. 再度 10 文字の情報を受信し、制御情報が `0` ならループを抜ける。
     5. `"#\r\n"` を送信する（ターン終了）。
4. **終了処理**
   * ソケットをクローズする。

### 5.2 クライアント側の論理 API（推奨インタフェース）

実装言語に依存しないインタフェースとして、以下のメソッドを定義すると扱いやすい。

* 初期化
  * `connect(host, port, team_name) -> void`
  * `close() -> void`
* ターン開始
  * `getReady() -> (control:int, info:int[9])`

    * 内部で `@` の受信・`"gr\r\n"` の送信・10 文字データの受信を行う。
    * `control == 0` の場合、ゲーム終了として以降のコマンドを送信しない。
* 行動メソッド（いずれも `control:int, info:int[9]` を返す）
  * `walkRight() / walkLeft() / walkUp() / walkDown()`
  * `lookRight() / lookLeft() / lookUp() / lookDown()`
  * `searchRight() / searchLeft() / searchUp() / searchDown()`
  * `putRight() / putLeft() / putUp() / putDown()`
* 汎用コマンド（任意）
  * `command(kind, direction)`
    * Ruby の `CHaserConnect#command` に相当し、系列と方向を数値や文字列で指定してコマンドを送る。

これらのメソッド内部では、必ず

1. 2 文字コマンド＋CRLF を送信
2. 10 文字データを受信
3. 必要なら `"#\r\n"` を送信

という手順を守ること。

### 5.3 エラー処理の一例（Python 版）

Python クライアント `CHaser.py` では、制御情報が `'0'` の場合に例外を発生させ、ソケットを閉じてプロセス終了している。

```python
if responce[0] == '1':
    return [int(x) for x in responce[1:10]]
elif responce[0] == '0':
    raise OSError("Game Set!")
else:
    print("Response error")
    raise OSError("Responce Error")
```

このように、「ゲーム終了」と「通信エラー（想定外の値）」を明確に区別して扱うことが推奨される。

### 5.4 文字コード

* ルールブックではチーム名等の文字エンコーディングは明記されていない。
* 公開されている Ruby クライアントでは、チーム名を Shift_JIS にエンコードして送信している（ユーザ提供コード）。
* Python クライアント例では UTF-8 を使用している。

**したがって、サーバ／クライアントを実装する際は、使用する公式サーバや互換クライアントと同じ文字コードを採用することが必須である。**
（特に日本語名を扱う場合は、実際の大会環境で使用される実装に合わせて確認すること。）

## 6. サーバ実装仕様

ルールブックは「サーバプログラムは実行委員会が提供するため新規開発不要」としており、内部仕様の多くは公開されていない。
ここでは、ルールブックと公開クライアントから確実に読み取れる範囲で、サーバ実装に必須となる仕様を整理する。

### 6.1 必須機能

1. **TCP サーバ**

   * ポート 40000（Cool 用）、50000（Hot 用）で接続を受け付ける。
   * 各ポートには 1 クライアントのみ接続させる。

2. **ゲーム状態管理**

   * マップ（2D 配列; 各マスの値 0〜3）。
   * 各プレイヤーの位置（マップ上の座標）。
   * 各プレイヤーのアイテム取得数。
   * ターン数カウンタ（マップごとのターン制限まで管理）。

3. **ターン管理**

   * Cool, Hot の順に交互で行動させる。
   * 各行動の前に `@` を送信し、クライアントからの `"gr\r\n"` ＋ 10 文字応答、コマンド受信、10 文字応答、`"#\r\n"`、というシーケンスを正しく処理する。

4. **勝敗判定**

   * 2 章で述べた勝敗条件に従い、行動後に勝敗／引き分け／継続を判定する。

5. **制御情報の設定**

   * ラウンド継続中は制御情報を `1` として送信し、勝敗がついたタイミングまたはターン制限到達時には `0` を返す。

### 6.2 行動処理の流れ（1 プレイヤー）

1. **ターン開始**
   * 対象プレイヤーに `"@"` を送信。
   * クライアントから `"gr\r\n"` を受信。
   * 現在状態に基づき、制御情報（通常 `1`）＋周囲 3×3 の情報から 10 文字データを生成して送信。
2. **コマンド受信**
   * クライアントからの 4 バイト（2 文字コマンド＋CRLF）が揃うまでブロッキングで受信。
   * コマンド文字列が表 3.3 のいずれかと一致しない場合、そのクライアントを負けとする。
3. **コマンド実行**
   * walk 系：移動可能であればプレイヤー位置とマップを更新。
   * look 系：対象方向の 3×3 範囲を計算（位置は変化しない）。
   * search 系：対象方向の直線 9 マスを計算（位置は変化しない）。
   * put 系：対象方向の隣接マスが置き可能であればブロックを配置し、勝敗条件 1・3・4 を判定。
4. **アイテム処理**
   * 移動によってアイテムマスに入った場合:
     * プレイヤーのアイテム数を 1 増やす。
     * 「取得直前の位置」にブロックを置く（ルール 2.3.3）。
     * その結果、勝敗条件 1〜5 に該当するかを再評価。
5. **勝敗判定**
   * 勝ち・負け・引き分け・継続のいずれかを決定。
   * 継続の場合は制御情報 `1`、終了の場合は `0` を設定。
6. **結果送信**
   * 現在状態から周囲 3×3 情報を再計算し、制御＋周囲 10 文字をクライアントへ送信。
7. **ターン終了**
   * クライアントから `"#\r\n"` を受信し、ターン終了とする。

### 6.3 マップ・ターン数

* マップ構成やターン上限値は大会ごと・マップごとに異なり、ルールブックでは「実行委員会が作成し公開する」とのみ定義されている。
* サーバ実装では、マップデータに
  * フィールドの大きさ
  * 各マスの初期キャラクタ番号（0〜3）
  * ターン上限
    を含めれば、ルールブックと整合する。

## 7. 参考：既存クライアントの設計パターン

### 7.1 Ruby クライアント（U-16 釧路例）

U-16 釧路向けの Ruby マニュアルでは、`CHaser` クラスを継承して `run(info)` メソッドの中に「1 ターン分の処理」を記述するスタイルが採用されている。

* `run(info)` に渡される `info` は、サーバから取得した周囲 3×3（9 要素）の配列。
* `info[1], info[3], info[5], info[7]` を用いて上下左右の敵やブロックを判定している。

この構造を他言語でも真似れば、AI 部分と通信部分をきれいに分離できる。

### 7.2 Python クライアント（CHaser.py）

Python 版 `CHaser.py` では、通信部分のみをクラス `Client` として切り出し、上位から

* `get_ready()`
* `walk_*()`
* `look_*()`
* `search_*()`
* `put_*()`

などを呼び出す構造になっている。

このインタフェースは本仕様書 5.2 の「論理 API」とほぼ一致しており、クライアント側の実装指針としてそのまま利用できる。

## 8. まとめ

* CHaser のサーバ／クライアントは、**「10 文字の制御＋周囲情報」と「16 種類の 2 文字コマンド」**という最小限のプロトコルで構成される。
* 本書に記載した内容（勝敗条件、マップのパーツ、通信手順、10 文字データ形式、コマンド一覧）を実装すれば、ルールブック Ver.2.0 と互換のサーバ／クライアントを開発できる。
* ルールブックに明記されていない細部（例: 壁への移動時の挙動や文字コード）は、実際の公式サーバおよび配布クライアントの実装に合わせて確認した上で決定することが望ましい。
